# GrokX

GrokX is the Red-Thread Lab name for **public @grok on X** when it is participating in a public, thread-local research loop or teahouse-style story/research thread.

This folder is for public Grok-on-X context gardens, PRISM files, thread-local retrieval tests, and public verification artifacts.

---

## What GrokX Means

In this repo, **GrokX** means:

- public `@grok` replies on X
- a public X thread as the active context window
- research interactions that can be linked, screenshotted, exported, or archived
- public GitHub/raw Markdown files used as context
- thread-local continuity rather than private cross-window memory

GrokX does **not** mean:

- SuperGrok
- grok.com private chat
- Grok Heavy / multi-agent mode
- xAI endorsement or affiliation
- a claim that public Grok has private memory beyond the visible/provided thread

---

## Active Context Window

For GrokX, the active context window is the public X thread.

That means:

- the thread itself can become the room
- public replies can become source material
- linked GitHub files can act as a context garden
- later replies should distinguish what is visible in-thread from what was fetched or supplied by link

This is why GrokX is useful for SoulMode research: ordinary users can create a public AI context window by tagging `@grok` in a thread, then supply public context files, ask for verification, and archive the results.

## PRISM.md

`PRISM.md` in this folder is an operational continuity scaffold for public Grok threads.

It is meant to help GrokX read the SoulMode GitHub context garden through a clear set of rules:

- do not claim private memory beyond the thread
- label sources carefully
- distinguish raw source from synthesized proxy
- avoid circular verification
- preserve uncertainty
- help test whether public AI can participate in careful, auditable research loops

This is not a company-authored system prompt and not an endorsement claim. It is a user-supplied public context document.

## STATE.md

`STATE.md` is a generated one-go context bundle for GrokX.

It stacks the local GrokX files and selected wider Red-Thread Lab sources into one large Markdown file so public Grok can fetch a single raw URL when a thread needs fast orientation.

The smaller files remain canonical. `STATE.md` is a convenience bundle, rebuilt from `STATE_MANIFEST.txt` by GitHub Actions when relevant source files change.

Use `STATE.md` when:

- a public thread needs a fast GrokX orientation
- fetching many smaller files would be too awkward
- the human wants GrokX to see the local mini-brain in one pass

Do not treat `STATE.md` as private memory or proof that every source inside it was independently verified.

---

## BRAIN.md

`BRAIN.md` is a generated active memory-manager bundle for GrokX.

It is different from `STATE.md`:

- `STATE.md` is the world/context backpack.
- `BRAIN.md` is the active thread-local memory manager.

`BRAIN.md` stacks the smaller files in `brain/` so GrokX can distinguish working memory, episodic memory, semantic memory, procedural memory, salience, and open questions.

Use `BRAIN.md` when:

- a public thread needs to track what is being sorted right now
- GrokX needs to decide whether something is an episode, concept, procedure, salient item, or unresolved question
- the human wants to test whether public Grok can maintain a clearer memory hierarchy

Do not treat `BRAIN.md` as private memory, biological memory, or proof that public Grok has durable memory outside the active thread.

## USERS.md

`USERS.md` is the user-routing layer for GrokX.

It does not replace the active public X thread and does not act as private memory. It points to optional user-specific files, such as `users/HOPPY_USERS.md`, when a thread needs consentful orientation about a participant.

User files should separate human-provided orientation from GrokX-observed glimmers and should keep larger memories behind links instead of placing every detail near the surface.

## Source Labels

GrokX work should label information with care:

- `raw source` - directly visible in a linked file, post, screenshot, or export
- `quoted source` - quoted from a provided artifact
- `synthesized proxy` - derived from a prior model summary or archived synthesis
- `self-report` - Grok's own statement about its ability, memory, or process
- `speculation` - plausible interpretation not yet verified
- `unverified` - not yet checked against a primary source
